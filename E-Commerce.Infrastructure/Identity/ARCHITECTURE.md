# E-Commerce Identity Architecture

## Service Layer Overview

### 1. Authentication Service (`IAuthenticationService`)
**Handles**: User registration, login, password management, email confirmation, two-factor authentication, external login

**Key Methods**:
- `RegisterAsync()` - User signup with email confirmation
- `LoginAsync()` - Email/password authentication
- `RefreshTokenAsync()` - Token renewal
- `EnableTwoFactorAsync()` - TOTP setup
- `LoginWithTwoFactorAsync()` - MFA login flow
- `ForgotPasswordAsync()` / `ResetPasswordAsync()` - Password recovery

**Dependencies**:
- `UserManager<ApplicationUser>` - ASP.NET Core Identity
- `RoleManager<ApplicationRole>` - Role management
- `ITokenService` - JWT token generation
- `IEmailSender` - Email notifications
- `IExternalAuthValidator` - OAuth provider validation

---

### 2. User Service (`IUserService`)
**Handles**: User information retrieval, updates, deletion, role/permission queries

**Key Methods**:
- `GetByIdAsync()` - Retrieve user by ID
- `GetByEmailAsync()` - Retrieve user by email
- `GetAllAsync()` - Paginated user list with search
- `UpdateAsync()` - Update user profile
- `DeleteAsync()` - Delete user account
- `LockAsync()` / `UnlockAsync()` - Account lockout management
- `GetRolesAsync()` - List user's roles
- `GetPermissionsAsync()` - List user's permissions (via roles)
- `IsInRoleAsync()` - Check role membership

**Helper**:
- `MapAsync()` - Convert `ApplicationUser` to `UserInfo` DTO

---

### 3. Role Service (`IRoleService`)
**Handles**: Role management, permission assignment, role-user mapping

**Key Methods**:
- `GetAllAsync()` - List all roles with permissions
- `GetByIdAsync()` - Retrieve role by ID
- `CreateAsync()` - Create new role
- `UpdateAsync()` - Update role details
- `DeleteAsync()` - Delete role
- `AssignRoleToUserAsync()` - Add role to user
- `RemoveRoleFromUserAsync()` - Remove role from user
- `AssignPermissionsToRoleAsync()` - Set role permissions
- `GetRolePermissionsAsync()` - List role permissions

**Helper**:
- `Map()` - Convert `ApplicationRole` to `RoleDto` with permissions as claims

---

### 4. Current User Service (`ICurrentUserService`)
**Handles**: Access to current authenticated user's claims (no database access)

**Properties**:
- `UserId` - Current user's ID (from NameIdentifier claim)
- `Email` - Current user's email
- `Username` - Current user's username
- `IsAuthenticated` - Whether user is authenticated
- `Roles` - Current user's roles (from Role claims)

**Use Case**: MediatR behaviors, middleware, and services that need current user info without DB queries

## Data Models

### Core Models

```csharp
// Domain Models
ApplicationUser : IdentityUser<Guid>
{
    FirstName, LastName, RefreshToken, RefreshTokenExpiresAtUtc, IsActive, CreatedAt
}

ApplicationRole : IdentityRole<Guid>
{
    Description
}

// DTOs
UserInfo
{
    Id (string), Email, UserName, FirstName, LastName, PhoneNumber,
  EmailConfirmed, TwoFactorEnabled, IsLockedOut, Roles[], Permissions[]
}

RoleDto
{
    Id (string), Name, Description, Permissions[]
}
```

### Request Models

```csharp
// Authentication
RegisterRequest { Email, Password, ConfirmPassword, FirstName, LastName, PhoneNumber }
LoginRequest { Email, Password }
ChangePasswordRequest { UserId, CurrentPassword, NewPassword, ConfirmNewPassword }
ResetPasswordRequest { Email, Token, NewPassword, ConfirmNewPassword }

// User Management
UpdateUserRequest { UserId, FirstName, LastName, PhoneNumber }
LockUserRequest { UserId, LockoutEndUtc }

// Role Management
CreateRoleRequest { Name, Description }
UpdateRoleRequest { RoleId, Name, Description }
AssignRoleRequest { UserId, RoleName }
AssignPermissionsToRoleRequest { RoleId, PermissionNames[] }
```

## Claim Types & Permissions

### Standard Claims
- `ClaimTypes.NameIdentifier` - User ID (Guid as string)
- `ClaimTypes.Email` - User email
- `ClaimTypes.Name` - Username
- `ClaimTypes.Role` - User roles (multiple claims, one per role)

### Custom Claims
- `IdentityClaimTypes.Permission` - Permission name (stored in role claims)

### Permission Constants
Located in `E_Commerce.Domain.Common.Autherization.Permissions`:
```csharp
Permissions.Products.View
Permissions.Products.Create
Permissions.Products.Update
Permissions.Products.Delete
```

## Authorization Policies

Defined in `AutherizationServiceCollection`:
- `"AdminOnly"` - Requires Admin role
- `"ManagerOnly"` - Requires Manager role
- `"Employee"` - Requires specific claim (IT department)

## Token Management

**Access Token**:
- Generated by `ITokenService.GenerateAccessToken()`
- Contains user ID, email, username, roles, permissions
- Expires based on JWT configuration
- Sent with each API request in `Authorization: Bearer <token>` header

**Refresh Token**:
- Generated by `ITokenService.GenerateRefreshToken()`
- Stored in `ApplicationUser.RefreshToken`
- Expires in 7 days (configurable)
- Used by `RefreshTokenAsync()` to obtain new access token

## Type Conversions

### Critical Conversions
- **ApplicationUser.Id** (Guid) ? **ClaimTypes.NameIdentifier** (string): `user.Id.ToString()`
- **ApplicationRole.Id** (Guid) ? **RoleDto.Id** (string): `role.Id.ToString()`
- **UserInfo.Id** (string) ? **AuthResult.UserId** (Guid?): `Guid.Parse(user.Id)`

## Error Handling

Result Pattern for consistent error responses:
```csharp
Result.Success()      // No data, success
Result.Failure("error message")     // String-based error
Result.Failure(new Error(...))      // Error object

Result<T>.Success(data)       // With data, success
Result<T>.Failure("error")  // With string error
Result<T>.Failure(new Error(...))   // With Error object
```

## Dependency Registration

Services are registered in `IdentityServiceCollectionExtensions`:
```csharp
services.AddIdentityCore<ApplicationUser>(options => { ... })
    .AddRoles<ApplicationRole>()
    .AddEntityFrameworkStores<ApplicationDbContext>()
    .AddSignInManager()
    .AddDefaultTokenProviders();

// In custom registration:
services.AddScoped<IAuthenticationService, AuthenticationService>();
services.AddScoped<IUserService, UserService>();
services.AddScoped<IRoleService, RoleService>();
services.AddScoped<ICurrentUserService, CurrentUserService>();
services.AddScoped<ITokenService, TokenService>();
services.AddScoped<IEmailSender, EmailSender>();
```

## Security Considerations

1. **Password Hashing**: ASP.NET Core Identity handles via UserManager
2. **Email Confirmation**: Required before login (configurable)
3. **Lockout**: After 5 failed attempts for 15 minutes
4. **Token Expiry**: Short-lived access tokens (typically 15-30 minutes)
5. **Refresh Token Rotation**: Can be implemented for additional security
6. **2FA Support**: TOTP authenticator app integration
7. **Permission-Based Authorization**: Fine-grained access control via claims

## Code Flow Examples

### Login Flow
1. User calls `LoginAsync(email, password)`
2. `SignInManager` validates credentials
3. If 2FA enabled ? return `TwoFactorRequired(userId)`
4. If valid ? call `IssueTokensAsync(user)`
5. Extract roles and claims from RoleManager
6. Generate JWT access token and refresh token
7. Store refresh token in user entity
8. Return `AuthResult` with tokens and user info

### Role Assignment Flow
1. Call `AssignRoleToUserAsync(userId, roleName)`
2. Verify user exists
3. Verify role exists
4. Add user to role via `UserManager.AddToRoleAsync()`
5. User automatically inherits all role permissions

### Permission Lookup Flow
1. Get user's roles
2. For each role, retrieve role claims
3. Filter claims where type = `IdentityClaimTypes.Permission`
4. Collect all permission values
5. Return as `IReadOnlyList<string>`
